# 路面異常検出システム 実現可能性検討報告書

## 1. 検討結果（要旨）

現時点の技術検討結果として、本システムは **条件付きでデモ実施可能** と判断する。

ただし、本判断は本番精度を保証するものではなく、現行 PoC データに基づく実現可能性評価である。特に、修繕済み路面を異常として誤検出しないことを最重要制約とし、再現率の向上は追加データの拡充とあわせて段階的に進める必要がある。

| 項目 | 現時点の評価 |
|---|---|
| 技術判定 | 条件付き GO |
| 優先すべき要件 | 修繕済み路面に対する偽陽性ゼロ |
| 現時点の主な課題 | 小型・遠方ポットホールの検出、修繕済み路面負例の不足 |
| 推奨方針 | 転移学習、hard negative 追加、保守的なしきい値判定を組み合わせる |
| デモ方針 | 高信頼検出を中心に、検出結果が後続処理へ連携できることを示す |

## 2. 検討目的

本書は、巡回車両のドライブレコーダ映像から路面異常を検出するシステムについて、現時点の PoC 結果、技術方針、作業計画およびリスクを整理するものである。

本検討の主目的は、検出モデル単体の精度確認ではなく、以下の処理が v0 デモとして成立するかを確認することである。

```
フレーム抽出 -> 異常検出 -> トリアージ -> 通知 -> 報告・記録
```

特に、修繕済み路面を異常として検出しないことを最重要制約とし、その制約下で明瞭なポットホールおよびひび割れの検出率を高める。

## 3. 前提条件と対象範囲

### 3.1 基本前提

| 項目 | 内容 |
|---|---|
| 入力 | ドライブレコーダ映像から上流処理で抽出された静止フレーム |
| 想定解像度 | 1080p H.264 を基本想定 |
| サンプリング | v0 は 1 fps |
| 推論方式 | サーバサイド推論。v0 では Cloud Run CPU を想定 |
| 学習方式 | 既存検出器および公開道路損傷データを起点とした転移学習 |
| v0 対象クラス | `pothole`、`crack` |
| 後続処理 | GPS、時刻、road_id を保持し、将来の複数フレーム集約に接続可能とする |

v0 ではフレーム単位の独立判定を許容する。ただし、本番運用では、同一異常が GPS 近傍で複数フレームに出現した場合のみ採用する集約フィルタを追加する想定である。

### 3.2 v0 対象範囲

| 区分 | 内容 |
|---|---|
| 検出 | ポットホール、未修繕のひび割れ |
| 抑制 | マンホール、路面標示、伸縮継手、排水グレーチング、修繕済みパッチ、封かん済みひび割れ |
| 出力 | 判定、クラス、信頼度、bbox、GPS、時刻、road_id、重大度スコア、対応目安 |
| 連携 | Pub/Sub イベント、JSON、注釈画像、地図・ダッシュボード表示、通知 webhook |
| 実行環境 | Cloud Run CPU、設定ファイルによるしきい値調整 |

### 3.3 v0 対象外

| 区分 | 内容 |
|---|---|
| 入力 | 動画ストリームの直接処理、車載リアルタイム推論 |
| クラス | 落石、倒木、表層劣化、工事中状態の専用分類 |
| 評価 | 本番精度保証、大規模車両群における SLA 確定 |
| 処理 | 高精度な深度推定、3D 路面復元、複数フレーム追跡 |

## 4. 受入目標

| 指標 | v0 デモ目標 | 本番目標 |
|---|---:|---:|
| 適合率（pothole + crack） | >= 0.90 | >= 0.97 |
| 明瞭なポットホールの再現率 | >= 0.50 | >= 0.80 |
| 修繕済み路面セット上の偽陽性 | **0** | < 0.1% |
| 1日分巡回映像の処理時間 | < 24時間 | 当日内、望ましくは < 6時間 |

上記のうち、修繕済み路面セット上の偽陽性ゼロを最優先の判定基準とする。再現率を上げるために修繕済み路面の誤検出を許容する運用点は、v0 デモでは採用しない。

## 5. これまでの技術検証

現行 PoC 画像を用いて、路面異常検出に必要な技術要素を段階的に確認した。ここでは、個別のモデル名や内部実験条件ではなく、実施した検証内容と得られた示唆を整理する。

### 5.1 実施した検証

| アプローチ | 結果 |
|---|---|
| 既存検出器による初期評価 | 一部の異常は検出可能。ただし修繕済み路面に偽陽性が発生 |
| しきい値調整のみ | 偽陽性ゼロを保つと再現率が大きく低下 |
| 汎用画像認識・領域抽出系の確認 | 緩い条件では検出数が増えるが、修繕済み路面の偽陽性を抑えきれない |
| 修繕済み負例を含めた追加学習 | 現行 PoC では偽陽性ゼロの運用点が得られるが、再現率は目標未達 |
| 人手修正ラベルによる追加学習 | 小規模データでは同一シーンの路面テクスチャに過適合する傾向 |

### 5.2 技術的示唆

現時点で得られた主な示唆は以下である。

- 修繕済み路面の誤検出は、単純なしきい値調整だけでは十分に抑制できない。
- 修繕済み路面を明示的な負例として学習に含めることが、偽陽性抑制に有効である。
- 一方で、正例データが少ない状態で追加学習を行うと、欠損形状ではなく路面テクスチャや同一シーン文脈を学習するリスクがある。
- そのため、正例の追加だけでなく、同一路面または近い路面テクスチャの負例を併せて追加する必要がある。
- v0 では、モデル単体の出力をそのまま採用せず、外部しきい値と保守的な判定ルールを併用する方針が妥当である。

ただし、これは実現可能性を示す初期根拠であり、本番精度を保証するものではない。現行データは規模が小さく、修繕済み負例の多様性も不足している。

## 6. 技術方針

### 6.1 基本方針

1. ゼロからの学習ではなく、既存検出器および公開道路損傷データを活用した転移学習を採用する。
2. 正例だけでなく、修繕済み路面、封かん済みひび割れ、マンホール、路面標示等の hard negative を重点的に追加する。
3. モデル出力をそのまま採用せず、異常スコアと修繕スコアを比較する保守的な判定ルールを設ける。
4. 信頼度しきい値および判定しきい値は設定ファイルで管理し、再学習なしで運用点を調整可能とする。
5. v0 は CPU 推論可能な軽量構成を優先し、モデル大型化よりもデータ品質と評価規律を優先する。

### 6.2 判定ルール

フレーム単位の判定は以下を基本とする。

```
if anomaly_score < MIN_DECISIVE_SCORE:
    verdict = normal
elif repair_score > anomaly_score + margin:
    verdict = normal
else:
    verdict = abnormal
```

このルールにより、明確な異常根拠がないフレーム、および修繕済み路面の可能性が高いフレームは normal 側に倒す。これは、修繕済み路面の偽陽性を最小化するための設計である。

## 7. データ計画

### 7.1 追加すべきデータ

| データ種別 | 目的 | 目安 |
|---|---|---:|
| 修繕済み路面 | 偽陽性抑制 | 約100枚以上 |
| 同一路面の before / after ペア | テクスチャ過適合の抑制 | 可能な限り収集 |
| 明瞭なポットホール | 基本再現率向上 | 継続追加 |
| 小型・遠方ポットホール | 低再現率領域の改善 | 継続追加 |
| マンホール、白線、継手、排水設備 | hard negative 強化 | 継続追加 |
| 雨天、夕方、逆光、濡れ路面 | 撮影条件の多様化 | 継続追加 |

### 7.2 データ運用上の注意

- 新しい正例シーンには、できる限り同一路面または近い路面テクスチャの負例を対応させる。
- 修繕済み路面を誤って pothole としてラベル付けしない。
- 判断が難しい箇所は、誤ラベルを避けるため未ラベルまたは normal 扱いとする。
- 保留評価セットは学習に使用しない。
- 評価では全体 mAP だけでなく、修繕済み路面 FP、明瞭ポットホール recall、hard negative FP を個別に確認する。

## 8. システム構成案

```
Dashcam video
    ↓
Frame extraction
    ↓
Frame metadata (GPS, timestamp, road_id)
    ↓
Detection service (Cloud Run CPU)
    ↓
Decision rule / thresholding
    ↓
Detection event + JSON + annotated image
    ↓
Triage / map / report / notification
```

v0 で新規に実装する中心部分は Detection service およびその入出力契約である。フレーム抽出、報告、通知、地図表示は既存系統に接続する前提で設計する。

## 9. 作業計画

| フェーズ | 作業内容 | 完了条件 |
|---|---|---|
| 準備 | 追加データ、ドラレコサンプル、メタデータ契約を受領 | 学習・連携に必要な入力が揃う |
| 第1週 | データセット再構成、追加学習、しきい値探索 | 修繕済み路面 FP = 0 の運用点を確認 |
| 第2週 | Cloud Run サービス化、Pub/Sub 入出力、JSON・注釈画像出力 | サンプルフレームから構造化検出結果が出力される |
| リハーサル | frames -> detector -> triage/report/notification を通す | デモ経路が end-to-end で動作する |
| デモ | 高信頼検出を中心にシステム動作を提示 | v0 受入目標に照らして結果を説明可能 |

デモでは、多数の境界的検出を出すよりも、少数でも説明可能な高信頼検出を提示する方針とする。

## 11. リスクと対応

| リスク | 影響 | 対応 |
|---|---|---|
| 修繕済み負例が不足 | 偽陽性ゼロを十分に評価できない | 追加データを優先収集。暫定デモでは制約を明記 |
| 再現率が 0.50 未達 | 明瞭な異常の見逃し | しきい値トレードオフを提示し、追加正例で改善 |
| 同一シーン過適合 | 修繕済み路面への誤検出 | before / after ペアと hard negative を追加 |
| 公開データとの分布差 | 実路面で精度低下 | 公開データは事前学習・補助用途に限定し、評価は実データ中心 |
| 連携仕様の確定遅延 | end-to-end デモ遅延 | 仮スキーマで先行実装し、後から正式仕様へ差し替え |
| ライセンス制約 | 本番展開時の制約 | 本番前に利用ライセンスを確認 |

## 12. 総合判定

現時点の技術判定は、**条件付き GO** とする。

理由は以下の通りである。

- 現行 PoC において、修繕済み路面に対する偽陽性を抑える方向性が確認できている。
- サーバサイド CPU 推論、外部しきい値、既存パイプライン連携という v0 構成は現実的である。
- 一方で、再現率および本番汎化については、追加データなしに強く主張できない。

したがって、デモに向けて開発を進める。ただし、追加データ、代表的なドラレコサンプル、連携仕様の確認を継続する必要がある。
